1. Executive Summary

VibeIDE is a GPU-native terminal orchestrator purpose-built for vibe coders — senior developers who run multiple AI agents in parallel across several projects and review their output as an air-traffic controller rather than a typist. The target user actively monitors 3–5 projects, uses voice as a primary input method, and demands sub-100ms latency on every interaction.

Phase 2 transitions from the Electron MVP to a native Rust application built on a crate-based architecture (wgpu + egui + alacritty_terminal) delivering 120 fps GPU rendering, integrated local voice (whisper-rs), and a WASM plugin system — targeting Linux, macOS, and Windows.

Key Decisions (Founder-Validated)

  1. Sidebar is sufficient for 3–5 projects — must be efficient and collapsible. No separate dashboard needed.
  2. Notifications must be BOTH visual AND auditory, configurable per event type. Non-negotiable requirement.
  3. Workspace templates are a key feature — ECC-based template presets shared across projects.
  4. File viewer as modal overlay is confirmed — not accessed often, but needed for confidence (Cursor/Windsurf parity).
  5. Voice is the PRIMARY input method, not secondary. Built-in local Whisper + cloud STT. No good voice app exists for Linux — this is a market differentiator. Cross-platform: Linux, macOS, Windows.

2. The Vibe Coder Persona

2.1 Who They Are

The vibe coder is a developer who has shifted from code writer to orchestrator. They describe what they want built in natural language and AI agents build it. Collins Dictionary named “vibe coding” the Word of the Year for 2025, signaling this is no longer a niche practice.

Demographics

  • Senior developers and technical founders (3–15+ years experience)
  • Comfortable in terminals; many have tmux/Zellij muscle memory
  • Run their own projects or lead small teams (1–5 people)
  • Work across 2–5 codebases simultaneously
  • Skew toward Linux and macOS power users
  • Early adopters who treat AI agents as multipliers, not toys

Identity: They think of themselves as architects, reviewers, and directors, not typists. The terminal is their command center, not a secondary tool.

2.2 Daily Workflow

A typical work session follows a four-phase cycle, repeated 3–5 times per day:

PhaseDurationActivity
Decompose5 minBreak work into parallel-safe tasks with clear boundaries
Launch3 minSpin up agents with focused prompts (voice or typed)
Review17+ minMonitor progress, validate scope, catch errors
Integrate10 minMerge branches, resolve conflicts, run tests

Key behavioral pattern: The vibe coder does NOT stare at one terminal continuously. They glance across multiple agent outputs, looking for signals: errors, completion messages, questions that need answers. The workflow is more like an air traffic controller than a pilot.

2.4 Pain Points (Ranked by Impact)

  1. Review is the bottleneck. Five agents completing simultaneously creates a cognitive pile-up. No tool helps triage which agent output to review first.
  2. No unified multi-project view. Developers lose ~23 minutes of productivity per context switch.
  3. Agents start fresh every time. Persistent memory across sessions is the #2 most-requested agentic IDE feature (RedMonk 2025).
  4. Terminal multiplexers have no agent awareness. tmux and Zellij treat every pane identically.
  5. Rate limits and cost opacity. No tool shows real-time cost/token usage across agents.
  6. Voice input is bolted on, not integrated. Voice is 5x faster than typing (220 WPM vs 45 WPM) but no tool makes voice a first-class dispatch mechanism.
  7. File browsing requires context switching. Reviewing agent work requires opening VS Code or running cat in a separate terminal.

2.5 What They Value

  • Speed. Sub-100ms latency on everything. GPU rendering. Instant startup.
  • Keyboard-first, mouse-optional. Everything reachable via keybinding or command palette.
  • Glanceability. Status at a glance without reading terminal output.
  • Minimal chrome. Maximum terminal real estate. Sidebars collapsible.
  • Session persistence. Close the app, reopen, everything exactly where it was.
  • Opinionated defaults, deep customization. Works perfectly out of the box; extensible when needed.

3. Core User Journeys

Journey 1: Starting a Work Session

Current behavior: Open terminal, cd to project, run tmux, manually open panes, start agents. Takes 2–5 minutes.

Optimal VibeIDE Experience

  1. Launch VibeIDE. Previous session restores instantly (layout, agents, scrollback).
  2. Dashboard view shows all projects with status summary.
  3. Click/select a project to enter its workspace. Agents resume where they left off.
  4. If starting fresh: Ctrl+Shift+P → “New Project” → select directory → choose agent template.

Design implication: VibeIDE needs a project-level abstraction above the current flat agent list. The sidebar should group agents by project.

Journey 2: Deep Work on a Single Project

Typical setup: 2–4 agents active. Common configurations: builder + reviewer (side by side), feature A + feature B + shell, or 2×2 grid.

Key Insight

The “needs input” state is the most time-critical notification. If an agent is blocked waiting for a yes/no, every second of delay wastes agent capacity. VibeIDE must make this state impossible to miss.

Design implication: Each terminal pane needs a thin agent status strip showing: agent type, status (running/idle/needs-input/error/complete), and elapsed time.

Journey 3: Multi-Project Monitoring

Current tools fail here. tmux windows per project means you can only see one at a time.

Optimal Experience

  1. Sidebar shows all projects with traffic-light summary: running normally, needs input, error, completed.
  2. Notification badges on project entries show unread events.
  3. Clicking a project switches the workspace area instantly (pre-rendered in background).
  4. Optional: Overview mode — compact view of all project workspaces (like macOS Mission Control for terminals).

Journey 4: File Exploration and Code Review

Key principle: VibeIDE is NOT an IDE. It does not need a full editor. A fast read-only viewer with diff support covers 90% of the use case.

Optimal Experience

  1. Ctrl+P opens fuzzy file finder scoped to current project.
  2. File opens in a read-only viewer pane (syntax highlighted, line numbers).
  3. Diff view via command palette or keybinding.
  4. Search across files (Ctrl+Shift+F) with ripgrep-speed results.
  5. File viewer is a temporary pane — dismisses with Escape.

Journey 5: Voice-Driven Workflow

Current state (March 2026): Claude Code shipped push-to-talk voice mode (March 3, 2026, 5% rollout). Codex shipped voice via Wispr engine. Voice is 5x faster than typing for natural language prompts.

Optimal VibeIDE Voice Experience

  1. Push-to-talk hotkey (configurable). Audio transcribed locally via Whisper.
  2. Transcript appears in a voice input bar. User reviews before sending.
  3. Voice commands are context-aware: focused agent pane receives the command; otherwise it is global.
  4. Command prefix detection: “vibeide” or “app” prefix routes to app commands; otherwise routes to focused agent.

Founder validation: Voice is the PRIMARY input method via Lotus Q. The founder wants built-in voice with local Whisper or cloud STT. No good voice app exists for Linux — this is a market differentiator.

What VibeIDE Must Do Better Than All Competitors

  1. Multi-project orchestration in one window — no competitor does this well
  2. Agent-aware terminal panes — status, notifications, and priority signals per pane
  3. GPU-native speed — 120fps, sub-5ms latency
  4. Voice as a first-class input — not bolted on; Linux differentiator

Detailed Competitor Notes

BridgeSpace

Tauri v2 + React 19 + xterm.js. Laggy on Linux due to WebKitGTK. Tries to be everything (terminal + editor + task board + swarm orchestrator). VibeIDE should be laser-focused on terminal orchestration at native speed.

claude-squad

Go TUI using tmux for session isolation and git worktrees for code isolation. No GPU rendering, no rich UI, no file viewer. VibeIDE has a strictly superior rendering surface.

Agents UI

Tauri + Rust, bundled Zellij. macOS-only. Auto-detected AI CLIs with branded icons is worth adopting.

Warp 2.0

Block-based command output is the killer feature. Rebranded as “Agentic Development Environment.” Core weakness: single-project focus, no multi-project orchestration.

Zellij

Mode-based interaction with discoverable keybindings. Floating panes pattern is perfect for file viewer, command palette, and voice transcript. WASM plugin system is a model for Phase 2.

5. Founder Validation

The following open research questions were validated directly by the founder (ND). These answers confirm or redirect key design decisions.

Q1: How many projects do you actively monitor in a single session?

Answer: 3–4 actively, can stretch to 5–7.

Design implication: Sidebar is sufficient — no separate dashboard view needed. Must be efficient and collapsible. Project list must scale gracefully from 3 to 7 entries with agent sub-items.

Q2: Notification preference — visual only or forced interruption?

Answer: Wants BOTH visual AND auditory feedback. Must be configurable per notification type. Strongly wants this.

Design implication: Implement a full notification configuration UI. Each event type (needs input, complete, error, build done) gets independent visual style, sound toggle, and focus-steal toggle. This is a non-negotiable requirement.

Q3: Do you use the same agent layout for every project?

Answer: Uses common plugin templates (ECC — Everything Claude Code) across projects. Starts with template presets. Workspace templates are a key feature.

Design implication: Implement a template system at ~/.vibeide/templates/. Ship with ECC default template. Templates define agent config, layout arrangement, and plugin set. “New Project” flow offers template selection.

Q4: How often do you read files during agent work?

Answer: Not often, but values having the capability for confidence (like Cursor/Windsurf). Confirms modal overlay approach.

Design implication: File viewer is a modal overlay (not persistent sidebar). Optimize for rare-but-important use. Keep the implementation lean — read-only, syntax highlighted, dismissible with Escape. Parity with Cursor/Windsurf in capability, not in prominence.

Q5: What is your most common voice command? Is voice primary or secondary?

Answer: Uses voice for EVERYTHING via Lotus Q (external app). Voice is the PRIMARY input method, not secondary. Wants built-in voice with local Whisper or cloud STT. Key insight: NO good voice app exists for Linux — this is a market differentiator. Building for Linux, macOS, AND Windows.

Design implication: Voice is promoted from “nice-to-have” to core feature. Integrate whisper-rs for local STT with cloud fallback. Voice command router must handle both app commands and agent prompts. The voice input bar is a primary UI element, not a secondary overlay. Linux voice support is a unique selling point — no competitor offers this.

8. Phase 2 Technical Architecture

Phase 2 uses a crate-based approach (NOT a WezTerm fork). Each responsibility is handled by a focused, battle-tested Rust crate composed into a unified application.

Core Crates

Core crates:
├── wgpu              → GPU rendering (WebGPU backend)
├── portable-pty      → PTY management (from WezTerm)
├── alacritty_terminal → VT parser + terminal state machine
├── cosmic-text       → Text shaping + glyph cache (System76)
├── egui              → UI chrome (sidebar, file explorer, notifications)
├── tokio             → Async I/O for PTY streams
├── notify            → File system watching for explorer
└── whisper-rs        → Local speech-to-text (voice engine)

Architecture Principles

  • Terminal grid and UI chrome share the same wgpu render pass
  • egui renders sidebar, file explorer, notifications on the same GPU context
  • Voice engine runs as a separate async task, feeds into command router
  • Project data model: Project → Agents → Terminals (hierarchical)
  • WASM plugin system for extensibility (wasmtime)

Voice Input Flow

Microphone → Whisper (local) or Cloud STT
  → Command router
    → If app command ("vibeide split vertical"): execute
    → If agent prompt: pipe to focused terminal's stdin
    → If ambiguous: show in command palette for confirmation
